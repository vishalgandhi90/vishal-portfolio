Building SwiftPaymentKit: A Protocol-Oriented Payment Framework

How I extracted patterns from years of payment gateway integrations into a reusable, open-source Swift library with async/await and actor-based concurrency.

Swift Open Source Payments SPM

The Problem

After integrating HyperPay, STC Pay, Tabby, Tamara, and Apple Pay into production apps over the years, I noticed a recurring pattern: every payment gateway has a different SDK, different initialization flow, and different callback mechanism. The app layer ends up with a tangled mess of gateway-specific code scattered across view controllers and managers.

What if every gateway could conform to a single protocol, and a central manager could route payments to the right one automatically?

The Core Protocol

The foundation of SwiftPaymentKit is the PaymentGateway protocol. Every gateway implementation conforms to this, providing a consistent interface regardless of the underlying SDK.

protocol PaymentGateway: Sendable {
    var identifier: String { get }
    var supportedMethods: [PaymentMethod] { get }

    func initialize(
        configuration: PaymentConfiguration
    ) async throws

    func createSession(
        amount: Decimal,
        currency: Currency,
        method: PaymentMethod,
        metadata: [String: String]
    ) async throws -> PaymentSession

    func processPayment(
        session: PaymentSession
    ) async throws -> PaymentResult
}

This design means you can swap gateways, add new ones, or run A/B tests between providers without touching the checkout flow.

Why Actors?

Payment processing involves mutable state: active sessions, retry counts, event callbacks. In a concurrent environment, this is a recipe for race conditions. Swift actors solve this elegantly.

public actor PaymentManager {
    private var gateways: [String: any PaymentGateway] = [:]
    private var activeSession: PaymentSession?

    public func register(
        _ gateway: any PaymentGateway
    ) {
        gateways[gateway.identifier] = gateway
    }

    public func processPayment(
        amount: Decimal,
        currency: Currency,
        method: PaymentMethod
    ) async throws -> PaymentResult {
        guard let gateway = gateways.values
            .first(where: {
                $0.supportedMethods.contains(method)
            })
        else {
            throw PaymentError
                .unsupportedMethod(method)
        }

        let session = try await gateway
            .createSession(
                amount: amount,
                currency: currency,
                method: method,
                metadata: [:]
            )
        activeSession = session
        return try await gateway
            .processPayment(session: session)
    }
}

The actor keyword ensures all access to gateways and activeSession is serialized automatically. No locks, no dispatch queues, no data races.

Payment Method Routing

One design decision I'm particularly happy with is automatic routing. When you register multiple gateways, the manager routes each payment to the gateway that supports the requested method.

let manager = PaymentManager()

// Register gateways
await manager.register(HyperPayGateway())
await manager.register(TabbyGateway())
await manager.register(ApplePayGateway())

// Routes to HyperPay automatically
let result1 = try await manager.processPayment(
    amount: 99.99,
    currency: .sar,
    method: .mada
)

// Routes to Tabby automatically
let result2 = try await manager.processPayment(
    amount: 299.99,
    currency: .sar,
    method: .tabby
)

Built-in Retry Logic

Network failures during payment processing are especially problematic. SwiftPaymentKit includes configurable retry logic with exponential backoff, so transient failures don't result in lost orders.

let config = PaymentConfiguration(
    environment: .production,
    merchantId: "merchant_123",
    timeout: 30,
    maxRetryCount: 3
)

await manager.configure(config)

Event Delegation

For analytics, logging, or UI updates, the library provides a PaymentEventDelegate protocol with lifecycle hooks. Default implementations are provided so you only override what you need.

protocol PaymentEventDelegate: AnyObject {
    func paymentDidStart(session: PaymentSession)
    func paymentDidComplete(result: PaymentResult)
    func paymentDidFail(error: PaymentError)
    func paymentDidCancel(session: PaymentSession)
}

What I Learned

  • Protocol-first design makes adding new gateways trivial. Conform to the protocol, register it, and the routing handles the rest.
  • Actors eliminate an entire class of bugs. Payment state management became dramatically simpler compared to manual locking.
  • Configuration over convention. Different apps need different retry counts, timeouts, and environments. Making these configurable from day one saved a lot of refactoring.
  • Test with mocks, not SDKs. The protocol design makes it easy to inject mock gateways in tests without importing any payment SDK.

Try It Out

SwiftPaymentKit is available as a Swift Package. Add it to your project with SPM and start integrating payment gateways with a clean, unified API.

// Package.swift dependency
.package(
    url: "https://github.com/vishalgandhi90/SwiftPaymentKit",
    from: "1.0.0"
)

Check out the GitHub repository for full documentation and examples.
