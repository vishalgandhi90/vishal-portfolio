Introducing CleanArchGen: Generate MVVM & VIPER Modules in Seconds

Why I built a CLI tool to eliminate the repetitive boilerplate of setting up Clean Architecture modules in iOS projects.

Swift Open Source CLI Clean Architecture

The Repetition Problem

Every time you add a new feature module to an iOS project following Clean Architecture, you end up creating the same set of files: a ViewModel, a View, a UseCase, a Repository, a DataSource, a Coordinator, and a DI container. For VIPER, it's even more: View, Interactor, Presenter, Entity, Router, and Protocols.

On a team of 15+ engineers, this boilerplate adds up. Inconsistent naming, forgotten files, and slightly different patterns across modules make the codebase harder to maintain. I wanted a tool that enforces consistency while saving time.

Usage

CleanArchGen is a command-line tool built with Swift ArgumentParser. Generating a module is a single command.

# Generate an MVVM module
$ cleanarchgen generate --name Login --pattern mvvm

  Created Login/LoginViewModel.swift
  Created Login/LoginView.swift
  Created Login/LoginUseCase.swift
  Created Login/LoginRepository.swift
  Created Login/LoginDataSource.swift

# Generate a VIPER module
$ cleanarchgen generate --name Profile --pattern viper

What Gets Generated

Each generated file follows conventions I've refined over years of building iOS apps at scale. Here's what an MVVM ViewModel looks like:

import Foundation
import Combine

final class LoginViewModel: ObservableObject {

    // MARK: - Published Properties
    @Published var isLoading = false
    @Published var error: Error?

    // MARK: - Dependencies
    private let useCase: LoginUseCaseProtocol
    private let coordinator: LoginCoordinator

    // MARK: - Init
    init(
        useCase: LoginUseCaseProtocol,
        coordinator: LoginCoordinator
    ) {
        self.useCase = useCase
        self.coordinator = coordinator
    }
}

The DI container wires everything together, and the test file comes pre-configured with mock dependencies.

Why a CLI Instead of Xcode Templates?

Xcode templates have several drawbacks that made them unsuitable for our needs:

  • Hard to version control. Xcode templates live in a system directory and aren't part of your project repo.
  • Team distribution is painful. Every developer needs to manually install them, and keeping them in sync is error-prone.
  • Limited logic. Templates can't conditionally include files or adapt based on parameters.
  • No CI integration. You can't run Xcode templates in a CI pipeline or pre-commit hook.

A CLI tool solves all of these. It's versioned with your project, installed via SPM, and can be integrated into automation scripts.

Custom Output Directory

By default, files are generated in the current directory. Use the --output flag to specify a target path.

$ cleanarchgen generate \
    --name Checkout \
    --pattern mvvm \
    --output Sources/Features/Checkout

Architecture Decisions

Swift ArgumentParser

I chose Apple's swift-argument-parser for the CLI framework. It provides automatic help generation, type-safe argument parsing, and a clean declarative API that makes adding new commands straightforward.

Template as Code

Templates are defined as Swift string interpolations rather than external template files. This means the compiler catches typos in template code, and refactoring is as simple as changing a function.

Colored Output

The built-in logger uses ANSI color codes for clear, readable terminal output. Success messages are green, file paths are highlighted, and errors stand out in red.

What's Next

  • Custom templates: Define your own patterns with a YAML or JSON config file.
  • SwiftUI vs UIKit: Flag to generate SwiftUI views or UIKit view controllers.
  • Xcode project integration: Automatically add generated files to the Xcode project.

CleanArchGen is open source and available on GitHub. Install it via SPM and start generating modules today.
